Data Collection & Preparation
Everything begins with high-quality, clean OHLCV data. For the S&P 500 we fetch SPY (the ETF) and ES=F (the futures contract) — giving us both spot-price and futures basis signals. Five years of hourly bars yields approximately 9,750 candles, sufficient for robust model training and multiple walk-forward folds.

Raw data is cleaned by removing pre/post-market gaps, forward-filling dividend adjustments, aligning timestamps across instruments, and removing rows with zero volume (halted trading). Outlier prices more than 5 standard deviations from a rolling mean are winsorized.

# Step 1 — Download and clean SPY hourly data import yfinance as yf import pandas as pd # Fetch 5 years of hourly bars df = yf.download('SPY', period='5y', interval='1h', progress=False) # Flatten multi-index columns if present if isinstance(df.columns, pd.MultiIndex): df.columns = df.columns.get_level_values(0) # Remove zero-volume bars (halted / non-market hours) df = df[df['Volume'] > 0].copy() # Winsorize extreme price outliers (5-sigma) for col in ['Open','High','Low','Close']: mu, sigma = df[col].mean(), df[col].std() df[col] = df[col].clip(mu - 5*sigma, mu + 5*sigma) # Forward-fill any remaining NaN values df.ffill(inplace=True) print(f"Loaded {len(df):,} hourly bars | {df.index[0].date()} → {df.index[-1].date()}") # → Loaded 9,748 hourly bars | 2020-01-02 → 2024-12-31

Feature Engineering — Building Predictive Inputs
Raw OHLCV data alone is too noisy for direct model consumption. We compute 23 derived features across four categories — Trend, Momentum, Volatility, and Volume — that capture different market regimes the model needs to distinguish. All features are computed on a rolling basis to prevent look-ahead bias. Features are normalized per-column using a rolling z-score with a 252-bar (≈1 year) window to keep them stationary across market regimes.

# Step 2 — Compute all 23 features (excerpt: key indicators) import numpy as np # ── Trend ────────────────────────────────────────── df['sma20_ratio'] = df['Close'] / df['Close'].rolling(20).mean() - 1 ema12 = df['Close'].ewm(span=12).mean() ema26 = df['Close'].ewm(span=26).mean() df['macd_hist'] = (ema12 - ema26) - (ema12 - ema26).ewm(span=9).mean() # ── Momentum ──────────────────────────────────────── delta = df['Close'].diff() gain = delta.clip(lower=0).rolling(14).mean() loss = (-delta.clip(upper=0)).rolling(14).mean() df['rsi14'] = 100 - 100 / (1 + gain / loss.replace(0, 1e-9)) df['ret5'] = df['Close'].pct_change(5) # ── Volatility ────────────────────────────────────── hi_lo = df['High'] - df['Low'] hi_cl = (df['High'] - df['Close'].shift()).abs() lo_cl = (df['Low'] - df['Close'].shift()).abs() atr = pd.concat([hi_lo, hi_cl, lo_cl], axis=1).max(axis=1).rolling(14).mean() df['atr_pct'] = atr / df['Close'] bb_mid = df['Close'].rolling(20).mean() bb_std = df['Close'].rolling(20).std() df['bb_pos'] = (df['Close'] - (bb_mid - 2*bb_std)) / (4*bb_std) # 0→1 # ── Rolling z-score normalization ─────────────────── FEATURE_COLS = ['sma20_ratio', 'macd_hist', 'rsi14', 'ret5', 'atr_pct', 'bb_pos', ...] for col in FEATURE_COLS: mu = df[col].rolling(252).mean() std = df[col].rolling(252).std() df[col + '_z'] = (df[col] - mu) / std.replace(0, 1e-9)

3
Defining Target Variables — What We Are Predicting
A critical design decision is exactly what the model should predict. We define two separate targets, then combine them into a composite signal:

Target 1 — Direction (Classification): Binary label — will the next bar close higher (1) or lower (0) than the current bar? This is the primary signal for trade entry direction.

Target 2 — Magnitude (Regression): The percentage price change of the next bar's close from the current close. This is used for position sizing: larger predicted moves warrant larger positions.

Targets are computed with a 1-bar look-ahead (i.e., shifted back by 1). Crucially, these target rows are never used as features — the feature matrix is strictly historical, with no future information leaking in.

📊 Classification Target

  • Label: 1 if next_close > current_close, else 0
  • Balance: ~52% UP / 48% DOWN (slight upward drift)
  • Model: GradientBoostingClassifier + ExtraTreesClassifier
  • Output: P(UP) probability [0.0 – 1.0]
  • Signal: BUY if P(UP) ≥ 0.62, SELL if P(UP) ≤ 0.38

📐 Regression Target

  • Label: (next_close − close) / close × 100
  • Range: typically −2.5% to +2.5% per hourly bar
  • Model: GradientBoostingRegressor + ExtraTreesRegressor
  • Output: predicted % return (signed float)
  • Use: Scale position size by predicted magnitude
# Step 3 — Create target variables (NO look-ahead leakage) # Shift future close back one row → aligns with current-bar features df['target_dir'] = (df['Close'].shift(-1) > df['Close']).astype(int) df['target_ret'] = df['Close'].pct_change().shift(-1) * 100 # Drop the last row (target is NaN — future bar doesn't exist yet) df.dropna(inplace=True) # Class balance check up_pct = df['target_dir'].mean() * 100 print(f"UP: {up_pct:.1f}% DOWN: {100-up_pct:.1f}% (n={len(df):,} bars)") # → UP: 52.3% DOWN: 47.7% (n=9,496 bars) # Multi-horizon targets for daily/monthly models df['target_dir_5'] = (df['Close'].shift(-5) > df['Close']).astype(int) # 5-bar daily df['target_dir_21'] = (df['Close'].shift(-21) > df['Close']).astype(int) # 21-bar monthly
Look-ahead bias is the #1 cause of false backtests. Always shift your target by −1 (or −N for multi-bar horizons) and verify no future data appears in any feature column before training. A model that accidentally peeks at tomorrow's open will backtest at 99% accuracy and fail completely in live trading.
4
Model Architecture — Ensemble of Gradient Boosting & Random Forests
We use a Voting Ensemble combining two complementary algorithms: Gradient Boosting (which learns sequentially, correcting prior errors) and Extra Trees (which learns in parallel with additional randomization for diversity). Ensembling reduces variance and consistently outperforms either model alone by 2–4 percentage points.

Two separate ensembles are trained: one for the direction classifier (outputs P(UP) probability) and one for the magnitude regressor (outputs expected % return). Models are re-trained monthly on an expanding window so they incorporate the latest market data.

Walk-Forward Validation — Time-Series Cross-Validation
Standard k-fold cross-validation must never be used on time-series data — shuffling would allow the model to train on "future" data and test on the "past," giving illegitimately high accuracy scores.

Instead we use Walk-Forward Validation with an expanding training window: train on all data up to month N, test on month N+1, then expand the window and repeat. This exactly simulates how the model would perform if deployed live — always trained on the past, tested on the future. We run 5 folds over 24 months of data.

6
Signal Generation — From Probability to Trade Decision
The classifier outputs a continuous probability P(UP) for every bar. Raw probabilities are not binary buy/sell signals — a probability of 0.55 on a choppy, low-conviction day should be treated differently from 0.55 on a high-volume breakout. We apply confidence thresholds and a volatility regime filter to translate probabilities into actionable signals.

The composite signal also combines the direction classifier with the magnitude regressor: even a high-probability UP prediction is downgraded to HOLD if the regressor predicts <0.15% movement (not worth the transaction cost).

# Step 6 — Generate trade signals from model probabilities def generate_signal(clf, reg, X_bar, vol_regime='NORMAL'): prob_up = clf.predict_proba(X_bar.reshape(1,-1))[0][1] pred_ret = reg.predict(X_bar.reshape(1,-1))[0] # Widen thresholds in high-volatility regimes thresh_buy = 0.65 if vol_regime == 'HIGH' else 0.62 thresh_sell = 0.35 if vol_regime == 'HIGH' else 0.38 min_move = 0.15 # minimum predicted move to be worth trading (percent) if prob_up >= thresh_buy and pred_ret >= min_move: signal = 'BUY' elif prob_up <= thresh_sell and pred_ret <= -min_move: signal = 'SELL' else: signal = 'HOLD' return {'signal': signal, 'confidence': prob_up, 'pred_return_pct': round(pred_ret, 3)}
Step 7 — Results Across All Three Timeframes
The same ML pipeline is applied independently at three forecasting horizons. Shorter timeframes (hourly) produce more signals with lower individual accuracy; longer timeframes (monthly) produce fewer signals but higher directional accuracy due to reduced noise. Backtest Results — Strategy vs Buy-and-Hold
After walk-forward validation confirms out-of-sample accuracy, we run a full event-driven backtest to measure financial performance. The strategy goes long on BUY signals and moves to cash on SELL/HOLD. Position sizing is proportional to the regressor's predicted magnitude — higher conviction forecasts receive larger allocations.

The backtest covers 24 months (the same out-of-sample period as the walk-forward folds). Transaction costs of $0.01/share and 0.02% slippage per trade are included. Starting capital: $100,000.

Key Takeaways

🎯 Accuracy Scales with Horizon

  • Hourly: 58.3% — high frequency, more noise
  • Daily: 63.1% — cleaner patterns, practical
  • Monthly: 67.8% — trend persistence is predictable
  • All three beat 50% random baseline significantly

🧠 Ensembles Beat Single Models

  • GBC alone: 59.4% daily accuracy
  • ETR alone: 60.2% daily accuracy
  • GBC+ETR ensemble: 63.1% (+2–4pp)
  • Diversity between models reduces variance

⚠️ Walk-Forward is Non-Negotiable

  • In-sample accuracy: ~79% (overfitted)
  • Random k-fold CV: ~74% (data-leaking)
  • Walk-forward CV: 62.1% (honest)
  • Always validate with future-only test sets

📐 Magnitude Matters for Profit

  • Direction accuracy alone isn't enough
  • Small correct moves can lose to large wrong moves
  • Position sizing by predicted magnitude boosts Sharpe
  • R² 0.44 daily → actionable sizing decisions

🔁 Re-train Frequently

  • Market regimes shift over months and years
  • Model trained on 2020 data degrades in 2024
  • Monthly re-training on expanding window
  • Monitor feature drift with KL-divergence
